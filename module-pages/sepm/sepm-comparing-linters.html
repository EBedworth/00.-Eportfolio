Comparing Linters

Given some (bad) Python code, run it through a variety of quality checking tools. Compare the output from each tool and compare the effectiveness of each.

Linter Outputs

The first tool that the code was quality checked with was PyLint. The output from PyLint was very detailed, and contained more areas to be improved than any of the other tools. PyLint also provides the name of the violation and the reference clause, if the relevant standard needs to be checked.

Next, I tested with PyFlakes. PyFlakes returned significantly fewer mistakes to the terminal. The reason appears to be that PyFlakes is not checking for errors such as whitespace at the end of a line, but is only really testing for coded errors. You can see that all of the points reported by PyFlakes are related to variables or imports:

PyCodeStyle seemed to be almost the opposite to PyFlakes, instead focusing on errors like whitespace usage and the number of lines between code. The output format is easily readbale and consistent with both of the other tools used so far.

PyDocStyle is different altogether from the other three tools, and is instead examing the docstrings used throughout the code. It gives specific information about missing docstrings but also about the tone and personness used in the language.

Tool Comparison

All of these tools serve the same overarching purpose, but with specific differences that make them useful. Of the four tools tested, I would say that PyLint appears to be the most 'all-rounded' tool, for checking most things in the code. If you were only able to use one tool, PyLint would probably be the one to select. However, it's long output could be a bit disheartening, especially with very large amounts of code, and might then encourage shortcuts to be taken.

PyFlakes and PyCodeStyle, when combined, perform a similar function to PyLint. It may be more user friendly to use one of these tools very often, and fix a small number of errors, than to use PyLint and try to fix everything. This would be particularly true if deviating from PEP-8 intentionally, such as to follow a particular in-house standard instead.

PyDocStyle performs a different function altogether, checking the docstrings and elements which will become the documentation. Whichever of the above tools was chosen, PyDocStyle would always also be chosen to be run alongside it.

Overall, it seems that the best approach to get full code coverage for errors is to use a combination of all of the tools, to ensure the code is checked against as many potential errors as possible.
